# My Streaming

A self-hosted video streaming server for your local network. Point it at a folder of `.mp4` files and watch them on any device вҖ” phone, tablet, or another computer вҖ” without copying a single byte.

## Quick Start

```bash
pip install -r requirements.txt
python server.py /path/to/videos
```

The server prints a URL and an ASCII QR code. Scan it or open the URL in any browser on the same LAN.

Omitting the directory defaults to the current working directory:

```bash
py server.py "E:\Developer\youtube-playground\download\yt-llm-translate"
```

## Requirements

- Python 3.8+
- FFmpeg (must be on `PATH`)
- NVIDIA GPU with CUDA drivers (for hardware-accelerated transcoding)

## Features

- On-the-fly HLS transcoding вҖ” no pre-conversion needed
- Resume playback from where you left off (per-video, stored in browser `localStorage`)
- Subtitle support вҖ” drop a `.srt` file next to the video, it will be converted to WebVTT and served as a `<track>` element
- Custom fullscreen button that rotates the player 90В° for landscape-only videos
- QR code for frictionless mobile access

## How It Works

### Why streaming instead of direct download?

A `.mp4` file can be hundreds of megabytes. If the browser downloaded the entire file before playing, you would wait minutes before seeing a single frame. Streaming solves this by slicing the video into short chunks (~10 seconds each) and feeding them to the browser one at a time. Playback starts almost instantly.

### HLS вҖ” HTTP Live Streaming

This project uses Apple's **HLS** protocol, natively supported by every modern browser. HLS has only two building blocks:

| File | Role |
|------|------|
| `.m3u8` playlist | A text manifest listing every segment URL in order, plus the duration of each segment |
| `.ts` segment | An actual chunk of video data, typically ~10 seconds long |

The browser reads the `.m3u8` to learn what segments exist, then fetches each one in sequence and plays them back-to-back with no gap.

```
User clicks a video
  вҶ’ browser fetches /stream/{id}/playlist.m3u8
  вҶ’ gets a manifest like:
      #EXTINF:10.0,
      segment_0.ts
      #EXTINF:10.0,
      segment_1.ts
      ...
      #EXT-X-ENDLIST
  вҶ’ browser fetches each .ts in order and plays them seamlessly
```

### Transcoding вҖ” why the original `.mp4` can't be used directly

The `.mp4` files on your disk may use arbitrary codecs. HLS requires **H.264 video** and **AAC audio** packed into `.ts` containers. FFmpeg re-encodes the video on the fly (hardware-accelerated via NVIDIA NVENC) and writes both the `.ts` segments and the `.m3u8` playlist to a `temp/` directory.

The server then serves the pre-computed static playlist immediately (with `#EXT-X-ENDLIST` present from the start), so the browser recognises the stream as **VOD** rather than **live**. This enables free seeking anywhere on the progress bar. If a segment has not yet been generated by FFmpeg, the server blocks the request for up to 60 seconds until the file appears.

### Seeking and the progress bar

When you drag the progress bar to, say, `2:30`, the browser's built-in HLS engine does a simple calculation:

```
150 s Г· 10 s per segment = segment 15
  вҶ’ request /stream/{id}/segment_15.ts
  вҶ’ start decoding from that point
```

None of this logic lives in the project's JavaScript вҖ” the `<video>` element's native HLS support handles seeking, segment selection, and buffering automatically.

### Subtitles

The scanner (`scanner.py`) looks for `.srt` files whose name starts with the same base name as the video. When found, the server converts them to WebVTT (`.vtt`), the subtitle format browsers understand, and injects a `<track>` element into the player page. The browser then matches subtitle cues to the current playback time automatically.

### Breakpoint resume

`player.js` saves the current playback position to `localStorage` every 5 seconds (and on pause/seek). When the same video is opened again, the stored timestamp is restored:

```
localStorage key: ms-pos-{video_id_hash}
```
